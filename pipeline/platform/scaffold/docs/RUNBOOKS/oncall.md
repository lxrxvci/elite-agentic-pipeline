# On-Call Runbook

## Alert Routing

Alerts generated by Prometheus are routed through Alertmanager.

### Local Development

```bash
# Start the full observability stack
docker compose up -d prometheus alertmanager grafana

# Open Alertmanager UI
open http://localhost:9093
```

### Receivers

| Receiver | Channel | Use Case |
|---|---|---|
| `slack` | `#alerts` | All alerts |
| `slack-critical` | `#incidents` | `severity=critical` alerts (duplicated from `#alerts`) |

### Configuring the Slack Webhook

1. Create an incoming webhook in your Slack workspace.
2. Export the URL as an environment variable:

   ```bash
   export SLACK_WEBHOOK_URL=https://hooks.slack.com/services/YOUR/WEBHOOK/URL
   ```

3. Restart Alertmanager:

   ```bash
   docker compose up -d alertmanager
   ```

In CI the Alertmanager configuration is validated with `amtool`.

## Common Alerts

### HighErrorRate

**Expression:** `rate(elite_request_duration_seconds_count{status_code=~"5.."}[5m]) > 0.01`

**Runbook:**
1. Check the backend logs in Grafana/Loki.
2. Look for recent deployments in GitHub Actions.
3. If the error rate is sustained, consider rolling back via the `rollback.yml` workflow.

### HighLatency

**Expression:** `histogram_quantile(0.95, rate(elite_request_duration_seconds_bucket[5m])) > 0.5`

**Runbook:**
1. Check database query performance in traces (Tempo).
2. Verify no N+1 queries were introduced.
3. Scale or roll back if needed.

### ServiceDown

**Expression:** `up{job="elite-backend"} == 0`

**Runbook:**
1. Verify the backend container/pod is running.
2. Check health endpoint: `curl http://localhost:8000/api/v1/health`.
3. Review recent deployment and restart if necessary.
