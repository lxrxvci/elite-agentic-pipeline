# RFC 0005: Photo-Driven Business Discovery

| Field | Value |
|---|---|
| Status | Proposed |
| Date | 2026-06-27 |
| Author | Tech Lead |

## Summary

Propose adding a photo-driven onboarding step to Foodcart SaaS. Owners snap or upload a photo of their food cart; the backend analyzes it with a multimodal LLM (Google Gemini), searches Google Places for the business, and pre-fills the onboarding form with name, address, phone, hours, website, Google Business Profile URL, Yelp, menu, and ordering links. The owner reviews and edits the proposals before publishing.

## Problem

Cycle 1 onboarding requires owners to locate and paste links to their existing digital presence (Google Business Profile, Yelp, menu, delivery platforms, social). This is friction at the exact moment we promise speed. Interview signals and drop-off data from the Links step suggest that many owners abandon onboarding to hunt for URLs. A photo of the cart is the most natural input — it already contains the business identity — and can shortcut the entire Links step.

## Proposed solution

### 1. New onboarding step: photo capture

Add a "Photo" step to the onboarding wizard between Identity and Links (flag-gated by `photo-onboarding-v1`). The step supports:

- File picker for existing images.
- Camera capture on mobile (`capture="environment"`).
- Client-side compression to ≤ 2 MB / 1920 px on the longest edge.
- Upload to Cloudflare R2 via a tenant-scoped presigned URL.

### 2. Object storage

- Cloudflare R2 bucket with S3-compatible API.
- Presigned upload URLs generated by the backend (`POST /uploads/presigned`).
- Object keys prefixed by tenant and site IDs: `{tenant_id}/{site_id}/{uuid}.webp`.
- Raw uploads are processed and then archived; only the chosen hero image remains publicly accessible.

### 3. Vision extraction adapter

- Extend the existing Gemini provider (`infrastructure/llm/gemini.py`) to accept a base64 image.
- Prompt Gemini to return a structured `VisionExtraction` object:
  - `business_name` (string, required)
  - `cuisine_type` (string, optional)
  - `visible_text` (list of strings, optional)
  - `location_hints` (list of strings, optional — city, neighborhood, landmark)
  - `confidence` (float 0–1)
  - `has_signage` (boolean)
- Gemini model: `gemini-2.0-flash` (multimodal, already wired).

### 4. Places enrichment adapter

- New `infrastructure/adapters/google_places.py` using the Google Places API (New).
- `places.searchText` with the extracted business name + location hints.
- `places.details` for the top result to retrieve:
  - name, formatted_address, phone_number, website
  - opening_hours / weekday_text
  - photos (optional)
  - url (Google Business Profile URL)
- Return a `PlaceDetails` Pydantic object.

### 5. Ingestion pipeline integration

- Extend `IngestionSourceType` with `PHOTO_VISION` and `GOOGLE_PLACES`.
- During onboarding:
  1. Create `UploadedImage` record.
  2. Run `PHOTO_VISION` ingestion job → `VisionExtraction`.
  3. Run `GOOGLE_PLACES` ingestion job using `VisionExtraction.business_name` + location hints → `PlaceDetails`.
  4. Run existing ingestion jobs (`WEBSITE`, `MENU_URL`, etc.) using the website from PlaceDetails.
  5. Merge normalized data into default content blocks as starting values.
- Proposals are surfaced in the onboarding form for owner review/edit before publish.

### 6. Hero image default

- The uploaded photo is used as the default hero image.
- UI includes a toggle to skip/change the hero image.
- If skipped, the existing template placeholder/fallback is used.

### 7. Feature flag and rollout

- Flag: `photo-onboarding-v1` (default `false`).
- Remove or default to `true` after pilot validation (≥ 70% upload completion, ≥ 80% name accuracy, no time-to-publish regression).

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| True Google Lens API | No public server-side API. Scraping consumer Google Lens violates ToS and is fragile. |
| Client-side OCR only | Less accurate than Gemini for noisy cart signage; does not provide business enrichment. |
| Skip Places API and rely only on Gemini extraction | Gemini can hallucinate contact details; Places API grounds the result in authoritative data. |
| Separate photo-only onboarding flow | Duplicates tenant/site creation logic and fragments the codebase. |
| Auto-publish without review | High risk of wrong identity; contradicts existing AI assistant guardrail pattern. |

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Gemini extracts wrong business name | Confidence score surfaced in UI; owner edits before publish; SPIKE-006 validates accuracy ≥ 70% before build. |
| Google Places API quota/cost | Cache results; per-tenant rate limits; manual fallback always available. |
| Food cart has no GBP/Places listing | Degrade gracefully to manual link entry; do not block onboarding. |
| Object storage abuse | File size/type limits; per-tenant upload quota; presigned URLs expire quickly. |
| Privacy / PII in photos | Strip EXIF; archive raw uploads; do not use publicly without owner preview. |
| Serverless timeout on image analysis | Compress images; monitor p95 latency; fallback to async job if needed. |

## Rollback plan

- Disable `photo-onboarding-v1` flag → onboarding reverts to URL-only flow.
- Delete or archive `UploadedImage` records and stored objects if needed.
- Replace generated hero blocks from revision snapshot if wrong image was published.

## Dependencies

- Cloudflare R2 bucket and S3-compatible credentials.
- Google Places API key and quota.
- Google Gemini API key (already configured).
- Existing `IngestionJob` framework and content block schema.
- Feature-flag infrastructure (`app/features.py`).
- SPIKE-006: photo→business-name accuracy validation.
