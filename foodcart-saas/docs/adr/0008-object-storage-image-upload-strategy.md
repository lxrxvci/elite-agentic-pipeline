# ADR 0008: Object Storage and Image Upload Strategy

| Field | Value |
|---|---|
| Status | Proposed |
| Date | 2026-06-27 |
| Author | Tech Lead |
| Deciders | Product Strategist, Backend Engineer, Security Champion |

## Context

Cycle 1 onboarding only accepts URLs for logos and hero images; there is no file upload path. The Photo-Driven Business Discovery bet (RFC 0005) requires owners to upload or capture a photo of their food cart during onboarding. We need a storage strategy that is cost-effective, tenant-isolated, secure, and consistent with the existing deployment target (Vercel + managed services).

## Decision

1. **Use Cloudflare R2** as the object-store for uploaded onboarding photos.
   - S3-compatible API, so the existing `boto3` dependency can be reused.
   - No egress fees, which matters if hero images are served from storage.
   - Aligns with the existing Cloudflare Registrar integration in the project.
2. **Frontend uploads directly to R2 via presigned URLs** generated by the backend.
   - Keeps large binary payloads out of the FastAPI request path.
   - Presigned URLs are short-lived (≤ 5 minutes) and tenant-scoped.
3. **Object keys are tenant-prefixed**: `{tenant_id}/{site_id}/{uuid}.webp`.
   - Enforces logical tenant isolation at the key level.
   - Simplifies lifecycle rules and cost attribution.
4. **Images are compressed client-side before upload**.
   - Target: ≤ 2 MB, longest edge ≤ 1920 px, WebP output when supported.
   - Reduces storage cost, egress, and serverless timeout risk.
5. **Raw uploads are processed and then archived**.
   - The processed/hero image is kept in a public or signed-URL-capable form.
   - The original raw upload is moved to a cold archive prefix after 24 hours and deleted after 30 days.
6. **The uploaded photo is used as the default hero image**, with a clear UI toggle to skip or replace it.
   - Owner sees the photo in preview before publish.
   - If skipped, the existing template fallback is used.
7. **EXIF metadata is stripped** and no biometric or people-recognition is performed.

## Consequences

### Positive

- Cost-predictable storage with no egress surprises.
- Reuses `boto3` and S3-compatible patterns; no new SDK needed.
- Presigned URLs keep the backend stateless and avoid large request bodies.
- Tenant-prefixed keys simplify access control and cleanup.

### Negative

- Adds an operational dependency on Cloudflare R2 credentials and bucket provisioning.
- Requires updating the Content-Security-Policy to allow the storage CDN origin.
- Requires a migration and repository layer for `UploadedImage` records.
- Image compression logic must be tested across mobile browsers and camera inputs.

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| AWS S3 | Egress fees would grow with published sites; R2 is cheaper for this read-heavy use case. |
| Vercel Blob | Ties storage to a single hosting vendor; R2 is portable and S3-compatible. |
| Store images as base64 in PostgreSQL | Unsustainable for photos; would bloat backups and slow queries. |
| Backend proxy uploads | Adds latency, timeout risk, and unnecessary egress through FastAPI. |
| Client-side only (no storage) | Cannot run server-side vision analysis or reuse the image as hero. |
