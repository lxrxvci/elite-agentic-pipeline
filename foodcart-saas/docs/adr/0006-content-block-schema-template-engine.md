# ADR 0006: Content Block Schema and Template Engine

| Field | Value |
|---|---|
| Status | Accepted |
| Date | 2026-06-24 |
| Author | Tech Lead |
| Deciders | Product Strategist, Frontend Engineer, Backend Engineer |

## Context

Each generated site must render the same seven sections across three reference templates. The content must be editable by owners, generated by onboarding, proposed by AI, and imported by ingestion. A stable, versioned schema is the glue that keeps these contexts from diverging.

## Decision

We will store each site as an ordered list of `ContentBlock` records in PostgreSQL JSONB:

- Each block has `block_type`, `schema_version`, `data`, and `sort_order`.
- Block types for Cycle 1: `hero`, `story`, `menu`, `locations`, `catering`, `contact`, `order_links`, `footer`.
- Backend validates `data` against versioned JSON Schemas using Pydantic v2.
- Frontend maps `(block_type, template_id)` to a React section component.
- Template-specific design tokens live in the frontend, not the database.
- Schema changes in Cycle 1 are additive only.

## Consequences

### Positive

- Shared contract simplifies AI, ingestion, and manual editing.
- JSONB balances flexibility with ACID persistence and queryability.
- Template engine keeps rendering concerns separate from content storage.
- Versioning gives a clear migration path as the product evolves.

### Negative

- Schema enforcement adds backend/frontend coordination overhead.
- Heavy reliance on JSONB may require monitoring as sites grow.
- Template-specific animations must not break block data assumptions.

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| HTML/rich-text blobs | Unsafe and impossible to patch deterministically. |
| Per-template schemas | Fragments ingestion and AI logic. |
| External headless CMS | Extra operational cost for MVP. |
